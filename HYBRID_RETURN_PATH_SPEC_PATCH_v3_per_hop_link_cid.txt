HYBRID RETURN PATH SPECIFICATION — PATCH v3
Per-Hop Link Circuit Identifiers (unlinkability fix)
Status: NORMATIVE ADDENDUM — supersedes conflicting text in v2 §4, §5, §7, §8, §9, §11, §13
Date: 2026-05-28

Purpose
───────
v2 reused one 16-byte circuit_id in cleartext on every return hop and in
every relay's CircuitTable. Colluding non-adjacent relays (or one operator
on multiple hops) can correlate return traffic and tie it to forward setup.

This patch replaces the global circuit_id with per-link identifiers:
  • Each hop stores a LOCAL inbound_link_cid (lookup key for packets it receives)
  • Each hop stores an LOCAL outbound_link_cid (value written into bytes 1-16
    when forwarding toward the client)
  • No single identifier appears on more than one hop-to-hop link

This is a wire-format and setup-material change only. AES-CTR layering,
nonce rules, magic framing, and SURB coexistence are unchanged.

A. THREAT MODEL (NEW — insert after §2 NON-GOALS)

A.1 What v2 global circuit_id broke

  Property lost: return-path hop unlinkability against non-adjacent collusion.

  Attack: Relay A and Relay C (not on the same link) observe the same
  circuit_id in cleartext on circuit packets and during forward setup.
  They learn immediately that traffic at A and traffic at C belong to the
  same logical stream, defeating the purpose of multi-hop return routing.

  Forward Outfox does not have this weakness because each hop sees fresh
  KEM ciphertext and hop-specific AEAD plaintext.

A.2 What this patch restores

  After v3, each relay only sees link-local identifiers on its ingress
  and egress circuit packets. Non-adjacent colluding relays cannot match
  streams by a shared cleartext ID.

  Adjacent hops on the same physical/link layer can always correlate
  (they share a link). That is expected and unavoidable.

A.3 What this patch does NOT fix

  • Timing / traffic-analysis resistance during streaming (still §2 non-goal)
  • Active tampering of AES-CTR payloads (still §10.4)
  • Replay after relay restart with persisted state (see §6 / §7 notes)

B. TERMINOLOGY (NEW)

  logical_circuit
      Client-local session handle. NEVER transmitted on the wire.
      Used only by the client to group keys, nonces, and re-establish
      state across forward requests. 16 random bytes recommended.

  inbound_link_cid
      16-byte cleartext lookup key on circuit packets this node RECEIVES
      on the return path. Unique per (relay, logical_circuit) pair.

  outbound_link_cid
      16-byte cleartext value this node WRITES into bytes 1-16 when
      forwarding a circuit packet toward the client.

  link binding invariant
      For adjacent return-path hops (U → V), U.outbound_link_cid MUST
      equal V.inbound_link_cid. Non-adjacent hops MUST NOT share either
      identifier.

C. PATCH §4 — CIRCUIT SETUP FIELDS (REPLACE v2 §4 fields)

Extended per-hop AEAD plaintext layout for circuit-bearing packets:

  routing              routing_size bytes
  timestamp            8 bytes
  flag                 1 byte              bitfield (unchanged from v2)
  inbound_link_cid     16 bytes            CID this hop expects on ingress
  key_seed             16 bytes            HKDF input for this hop's key
  next_hop             routing_size bytes  return-path next hop (toward client)
  outbound_link_cid    16 bytes            CID written on egress toward client
  ttl                  2 bytes             circuit lifetime (uint16 BE)
  [inner_header]

Per-hop circuit overhead (routing_size=16):
  16 + 16 + 16 + 16 + 2 = 66 bytes  (+16 vs v2's 50 bytes)

Header budget: OutfoxParams.header_sizes(circuit_setup=True) MUST use 66
bytes per hop, not 50.

Key derivation (VERSION BUMP — v2 used circuit_v1 + global id):

  circuit_key = HKDF-SHA256(
    ikm  = key_seed,
    salt = inbound_link_cid,
    info = "circuit_v2",
    len  = 16
  )

  Rationale: table lookup salt matches derivation salt; no global id exists.

Registration during forward processing (REPLACE v2 register_circuit args):

  register_circuit(
    inbound_link_cid,
    circuit_key,
    next_hop,
    outbound_link_cid,
    ttl
  )

Client generation during packet_create_repliable() (REPLACE v2 §9.1a):

  a. Generate logical_circuit (16 bytes, client-local only, never on wire)
  b. For each return-path hop H (exit first, then relays toward client,
     then client as terminal):
        - inbound_link_cid_H  (random 16 bytes)
        - key_seed_H          (random 16 bytes)
        - outbound_link_cid_H (random 16 bytes, except see binding below)
  c. Enforce link binding invariants along the return path:
        exit.outbound_link_cid           = relay_n.inbound_link_cid
        relay_i.outbound_link_cid        = relay_{i-1}.inbound_link_cid
        last_relay.outbound_link_cid     = client.inbound_link_cid
     (Use actual reply-path ordering, not forward-path indices.)
  d. Derive circuit_key_H = HKDF(key_seed_H, salt=inbound_link_cid_H,
     info="circuit_v2") for each hop
  e. Embed per-hop setup in the corresponding forward AEAD routing
  f. Client stores:
        logical_circuit
        client.inbound_link_cid          (receive lookup key)
        exit_key                         (innermost decrypt key)
        relay_keys[]                     (retained for re-establish only)
        nonce_watermark, corruption_count

DEPRECATION: v2 field name circuit_id in setup material and on the wire
is REMOVED. Implementations MUST NOT send one global id to all hops.

D. PATCH §5 — CIRCUIT PACKET FORMAT (REPLACE v2 §5 bytes 1-16 description)

Wire layout unchanged except field semantics:

  Byte offset   Field
  ──────────    ─────────────────────────────────────
  0             type = 0x01 (circuit packet)
  1-16          link_cid (16 bytes)   ← renamed from circuit_id
  17-24         nonce (8 bytes, big-endian counter)
  25-end        encrypted region (unchanged)

  link_cid is HOP-LOCAL:
    • On the link U → V, sender U MUST set link_cid = V.inbound_link_cid
    • Receiver V looks up CircuitTable[link_cid]
    • link_cid MUST NOT be reused across non-adjacent links

  Total packet size, magic, inner_len, padding rules: unchanged from v2.

E. PATCH §7 — RELAY PROCESSING (REPLACE v2 circuit_packet_process steps)

When a relay receives type 0x01:

  1. Read link_cid from bytes 1-16
  2. Look up entry = CircuitTable[link_cid]
     Entry fields:
       circuit_key, next_hop, outbound_link_cid,
       high_watermark, expiry, last_active
  3. If not found or expired: drop (unchanged rationale from v2)
  4. Read nonce from bytes 17-24
  5. If nonce <= high_watermark: drop (replay/reorder)
  6. high_watermark = nonce
  7. Decrypt bytes 25 onward: AES-CTR(circuit_key, iv=nonce||0x00*8)
  8. Reconstruct outbound packet:
       type(1) + outbound_link_cid(16) + nonce(8) + decrypted_payload
     *** HEADER REWRITE RULE ***
     Bytes 1-16 MUST be replaced with entry.outbound_link_cid before
     forwarding. Never forward the ingress link_cid downstream.
  9. Forward reconstructed packet to entry.next_hop
  10. Update last_active for TTL / LRU

Exit node (first sender on return path):

  • Does NOT perform step 1-2 lookup when creating packets.
  • Uses outbound_link_cid stored in its forward setup (equals first
    return relay's inbound_link_cid).
  • Creates packet per §3 crypto layering, then sets bytes 1-16 to that
    outbound_link_cid.

Self-healing (circuit_self_heal): unchanged — only when entry exists.

F. PATCH §8 — CLIENT DECRYPTION (REPLACE v2 lookup key)

Client circuit state (REPLACE v2 circuit_keys dict keyed by circuit_id):

  pending_circuits: {
    client_inbound_link_cid: {
      "logical_circuit": bytes,        # local only
      "exit_key": bytes,
      "relay_keys": [bytes, ...],      # retained, not used on receive
      "nonce_watermark": int,
      "corruption_count": int,
    }
  }

On receiving a circuit packet:

  1. Read link_cid from bytes 1-16
  2. Look up pending_circuits[link_cid]  (NOT a global circuit_id)
  3. Read nonce from bytes 17-24
  4. If nonce <= nonce_watermark: discard
  5. nonce_watermark = nonce
  6. Decrypt bytes 25 onward with exit_key only (unchanged)
  7-12. Magic check, corruption counter, re-establish: unchanged from v2

Re-establishment creates a NEW logical_circuit and NEW link CIDs for
all hops. Old link_cids MUST NOT be reused.

I. WORKED EXAMPLE (NEW)

Return path: Exit → Relay B → Relay A → Client
(Forward path was Client → Relay A → Relay B → Exit)

Client generates (random except bindings):

  logical_circuit = L (never on wire)

  Client:   inbound = C_cli
  Relay A:  inbound = C_A_in,  outbound = C_A_out  where C_A_out = C_cli
  Relay B:  inbound = C_B_in,  outbound = C_B_out  where C_B_out = C_A_in
  Exit:     inbound = C_E_in,  outbound = C_E_out  where C_E_out = C_B_in
            (exit inbound unused on receive; may be set to 0x00..00)

Forward setup embeds each hop's fields in its AEAD routing.

Streaming packet from Exit toward Client:

  Exit sends:    [0x01 | C_B_in | nonce | Enc(...)]
  Relay B recv:  lookup C_B_in, peel, rewrite → [0x01 | C_A_in | nonce | ...]
  Relay A recv:  lookup C_A_in, peel, rewrite → [0x01 | C_cli  | nonce | ...]
  Client recv:   lookup C_cli, decrypt exit layer, verify magic

Colluding Relay A + Exit: see C_B_in and C_E_out (= C_B_in) — adjacent link
  knowledge only.

Colluding Relay A + Relay B: see shared link CID on A↔B segment — expected.

Colluding Relay A + Exit (non-adjacent in some topologies): see C_A_* vs
  C_B_in / C_E_out — different values, no direct correlation by link_cid.

J. INTEROPERABILITY / MIGRATION

  • wire format byte layout unchanged; semantics of bytes 1-16 change
  • HKDF info string bumps circuit_v1 → circuit_v2
  • setup material grows +16 bytes per hop
  • v1 and v2 implementations MUST NOT interoperate on circuit packets

Migration strategy:

  Phase 2.5 (insert before Phase 3 in §13):
    - Replace global circuit_id with inbound/outbound link CIDs in setup
    - Extend CircuitTable with outbound_link_cid
    - Implement relay header rewrite (§E step 8)
    - Bump derive_circuit_key to circuit_v2 salt=inbound_link_cid
    - Add collusion regression test: same logical stream, non-adjacent
      relay views must show different link_cid values

Dual-read: NOT required for link_cid (breaking semantic change). Clients
and relays must upgrade together for circuit streaming.

L. TEST VECTORS (NORMATIVE FOR IMPLEMENTERS)

L.1 Setup binding

  Given return path [Exit, B, A, Client], client-generated tables MUST
  satisfy:
    Exit.outbound  == B.inbound
    B.outbound     == A.inbound
    A.outbound     == Client.inbound

L.2 Relay rewrite

  If relay B receives packet P with P[1:17] == B.inbound, forwarded
  packet P' MUST have P'[1:17] == B.outbound and P'[17:25] == P[17:25].

L.3 Collusion observability

  In a 4-hop return simulation, record all link_cid values seen at each
  node. For nodes X and Y where Y is not X's next_hop or prev_hop on the
  return path, the sets of observed link_cids MUST be disjoint for a
  single logical_circuit stream.
